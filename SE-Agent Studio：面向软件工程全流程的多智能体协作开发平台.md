# SE-Agent Studio 课程设计方案（现状同步版）

## 一、项目基本信息

- 项目名称：SE-Agent Studio：面向软件工程全流程的多智能体协作开发平台
- 项目类型：软件工程课程设计 / 综合实践项目
- 项目定位：基于 CrewAI 与大语言模型能力，构建一个支持技术设计、代码交付、过程追踪与结果下载的 Web 平台
- 当前形态：固定工作流驱动的多智能体协作平台，已支持 `technical_design_v1` 与 `delivery_v1`
- 适用对象：软件工程、计算机科学与技术、人工智能等相关专业学生

## 二、项目概述

随着大语言模型和智能体系统的发展，软件开发过程正在从“单一工具辅助”逐步走向“多角色协作式智能开发”。传统课程设计项目多集中于管理系统、论坛系统、电商系统等典型 CRUD 场景，虽然实现难度适中，但在创新性、工程完整性和展示效果方面往往较为有限。

SE-Agent Studio 的目标不是重新发明底层多智能体框架，而是面向软件工程教学与展示场景，在 CrewAI 之上实现一个可运行、可观测、可交付的平台。平台内置产品经理、软件架构师、后端架构师、前端开发工程师、AI 工程师和 API 测试工程师等角色，围绕用户输入的项目需求自动组织协作。

和早期“只产出文档”的方案相比，当前系统已经扩展为双工作流结构：

1. `technical_design_v1`：生成结构化技术设计产物。
2. `delivery_v1`：生成可运行 starter 项目、执行基础验证，并提供项目压缩包下载。

因此，本项目的当前定位已经从“多智能体技术设计平台”升级为“多智能体软件工程协作与 starter 交付平台”。

## 三、研究背景与意义

### 3.1 研究背景

当前，AI 编程助手已经能够在代码补全、错误修复和文档生成等方面提供支持，但多数工具仍以“单轮问答”或“单角色助手”为主，缺乏软件工程视角下的角色分工与协作流程。现实的软件开发工作通常由产品、架构、开发、测试等多个角色共同参与，各角色关注点不同，产出形式也不同。如果能够将这些角色抽象为多个可协同工作的智能体，并将其组织为统一的软件工程工作流，将有助于提高软件开发效率，并为教学场景提供直观的实验平台。

与此同时，CrewAI 等开源智能体框架已经提供了较成熟的 Agent、Task、Crew、Flow 等机制，使开发者能够更高效地实现多智能体协作流程。因此，在课程设计场景中采用成熟框架完成平台集成，比完全自研底层编排引擎更具现实可行性，也更有助于把有限时间投入到业务流程设计、结果管理和系统展示上。

### 3.2 项目意义

本项目的研究与实现具有以下意义：

1. 体现软件工程全流程思想。项目覆盖需求、设计、交付、验证和结果管理等多个环节，较好地符合软件工程课程的教学目标。
2. 具备较强的前沿性。相比传统信息管理系统，本项目融合了大语言模型、CrewAI 多智能体协作和软件工程流程编排，主题更贴近当前 AI 软件工程发展方向。
3. 强调工程实践能力。项目涉及前后端分离、接口设计、数据库建模、运行状态追踪、结构化输出、项目打包与交付等内容，能够训练学生的综合开发能力。
4. 便于成果展示。系统演示时不仅能展示“输入需求后自动生成多类软件工程产物”，还可以展示 starter 项目生成、自动验证和压缩包下载，答辩效果更强。

## 四、项目目标

本项目的总体目标是设计并实现一个基于 Web 的多智能体协作平台，使其能够围绕一个软件项目需求，基于 CrewAI 自动组织多个软件工程角色完成任务处理，并形成结构化成果输出或可运行交付结果。

具体目标如下：

1. 实现用户鉴权、项目管理、角色配置和流程模板的基础管理能力。
2. 支持用户提交项目需求，并发起固定工作流运行。
3. 支持技术设计流与代码交付流两种典型软件工程协作路径。
4. 保存每次流程运行的上下文、输入输出、执行状态、耗时和异常信息，形成完整协作记录。
5. 将需求规格、架构方案、技术设计、代码文件、验证结果等产物统一展示与导出。
6. 支持项目交付压缩包下载，提升最终结果的可接收性。
7. 支持账号级与角色级模型配置，提高多智能体运行的灵活性。

## 五、需求分析

### 5.1 用户角色

当前系统主要包含以下两类用户：

1. 普通用户：可登录系统、创建项目、提交需求、发起工作流、查看结果并下载交付包。
2. 管理员：可维护角色模板、流程模板和系统级配置，并查看平台当前支持的工作流与角色状态。

### 5.2 功能需求

#### 5.2.1 鉴权与项目管理模块

系统需要支持用户登录、项目创建、项目编辑、项目列表展示、项目详情查看和项目删除。用户可以围绕不同题目创建多个项目，并在项目中保留历史运行记录和交付结果。

#### 5.2.2 角色模板与流程模板模块

系统内置多个软件工程角色，包括产品经理、软件架构师、后端架构师、前端开发工程师、AI 工程师和 API 测试工程师。管理员可查看各角色的职责说明、默认模型和启用状态，并维护流程模板的启用状态与配置。

#### 5.2.3 需求录入与工作流启动模块

用户在项目中输入自然语言需求后，系统应支持以固定工作流方式发起运行。当前默认以 `delivery_v1` 为主要启动路径，也保留 `technical_design_v1` 作为设计产物工作流。

#### 5.2.4 多智能体协作与执行模块

平台基于 CrewAI 的 Agents、Tasks、Crews 和 Flows 机制组织协作流程。首版系统采用“固定流程模板 + 有限参数配置”的方案，支持以下两种典型方式：

1. 串行协作：前一个角色的输出作为后一个角色的输入，例如“需求分析 -> 架构设计 -> 交付移交”。
2. 有限并行协作：在某些阶段允许多个角色围绕同一需求并行生成结果，例如前后端并行生成代码包或设计产物。

系统当前不追求可视化自由编排，而是优先保证流程稳定性、结果一致性和演示效果。

#### 5.2.5 执行追踪与上下文管理模块

系统需要记录每次流程运行的输入内容、阶段状态、任务输出、执行时间、异常信息以及与项目关联的上下文数据，便于用户回溯结果来源，增强系统的可解释性和可维护性。

#### 5.2.6 产物中心与交付模块

系统将不同角色生成的内容进行结构化整理，形成项目产物集合，包括但不限于：

- 需求规格
- 架构方案
- 后端设计或后端代码包
- 前端设计或前端代码包
- AI 集成方案
- 测试计划或集成验证结果
- 最终交付移交说明

对于 `delivery_v1`，系统还应支持：

1. 将生成文件写入工作区。
2. 执行基础验证。
3. 打包最近一次成功交付结果并提供 zip 下载。

#### 5.2.7 模型配置模块

系统需要支持用户自己配置兼容 `OpenAI API` 的模型服务，包括：

1. 账号默认 `provider / base_url / api_key / model`
2. 角色独立 `provider / base_url / api_key / model`
3. 读取 `/models` 获取真实可用模型列表

#### 5.2.8 运行监控与分析模块

系统应在运行页展示阶段进度、事件时间线、任务详情、验证结果和失败原因，并为交付场景提供明确的下载入口。

### 5.3 非功能需求

#### 5.3.1 性能与启动需求

- 本地开发应支持最小依赖启动
- 普通查询接口应具备较快响应
- 长时任务应支持异步查询与流式状态更新
- 本地默认运行模式不应强依赖 `Redis` 和 `MySQL`

#### 5.3.2 可用性需求

- 系统界面应便于教师和学生快速上手
- 关键路径尽量收敛到“创建项目 -> 输入需求 -> 启动协作 -> 下载结果”
- 运行细节用于排障，不应压过最终结果入口

#### 5.3.3 安全性需求

- 支持基本的登录认证与权限控制
- 用户数据、项目数据和运行结果应与其他用户隔离
- 大模型 API Key 不得明文展示
- 管理类接口仅对管理员开放

#### 5.3.4 可维护性需求

- 前后端代码结构清晰，模块划分明确
- 后端采用分层设计，并将 CrewAI 集成逻辑与业务逻辑隔离
- 工作流、角色模板、模型配置应便于后续扩展
- 运行状态、事件和产物应统一持久化，便于排障与演进

## 六、总体架构设计

### 6.1 架构选型

本项目采用“前后端分离 + 模块化单体后端 + CrewAI 运行时集成”的总体方案。前端负责用户交互与结果展示，后端负责业务逻辑、流程模板管理、任务执行编排、数据存储以及大模型接口调用。

采用该方案的主要原因如下：

1. 课程设计周期有限，基于成熟框架进行二次集成比自研编排引擎更高效。
2. 系统当前用户规模较小，模块化单体即可满足开发和展示需求，没有必要引入微服务运维复杂度。
3. CrewAI 已提供较成熟的多智能体流程组织能力，项目团队可以把精力集中在教学场景适配、执行追踪、交付打包和结果展示等更有价值的部分。

### 6.2 系统总体结构

```text
+-------------------+        +-------------------------------+
|     Web 前端      | <----> |         FastAPI 后端          |
| Vue3 + ElementPlus|        | Auth / Project / Run / LLM    |
+-------------------+        +---------------+---------------+
                                             |
                    +------------------------+------------------------+
                    |                                                 |
                    v                                                 v
        +---------------------------+                  +------------------------+
        |        应用服务层         |                  |       数据存储层       |
        | 项目/运行/产物/打包/配置   |                  | SQLite / MySQL(可选)   |
        +-------------+-------------+                  +------------------------+
                      |
                      v
        +---------------------------+
        |    CrewAI 集成运行层      |
        | Agents / Tasks / Crews /  |
        | Flows / Guardrails / LLM  |
        +-------------+-------------+
                      |
                      v
        +---------------------------+
        | OpenAI兼容接口 / DeepSeek |
        | Qwen / 其他兼容模型服务    |
        +---------------------------+
```

### 6.3 后端模块划分

后端按照职责划分为以下几个模块：

1. 用户认证模块：处理登录、身份校验与权限控制。
2. 项目管理模块：处理项目创建、详情、删除和交付打包。
3. 角色与流程模板模块：维护 Agent 配置、提示词模板、流程模板和启用状态。
4. CrewAI 集成模块：负责 Agent、Task、Crew、Flow 的构建、执行与调度。
5. 运行记录模块：保存流程运行状态、阶段输出、异常信息和事件。
6. 结果管理模块：整理并展示各角色生成的结构化产物。
7. 模型配置模块：维护账号级和角色级运行时模型配置。

### 6.4 前端模块划分

前端当前主要包含以下页面：

1. 登录页面
2. 项目列表页面
3. 项目详情页面
4. 运行监控页面
5. 产物中心页面
6. 模型配置页面
7. 管理配置页面

## 七、核心业务流程设计

### 7.1 双工作流主流程

当前系统围绕两个固定工作流展开：

#### 技术设计流 `technical_design_v1`

```text
需求结构化
  -> 架构设计
  -> 并行设计
      -> 后端设计
      -> 前端设计
      -> AI 集成设计
  -> 测试方案
  -> 一致性评审
```

#### 代码交付流 `delivery_v1`

```text
交付需求
  -> 方案设计
  -> 并行交付
      -> 后端代码包
      -> 前端代码包
  -> 集成验证
  -> 交付移交
  -> 项目压缩包下载
```

### 7.2 典型交付链路

建议系统在答辩或演示中优先展示以下链路：

1. 用户登录并进入项目空间。
2. 创建项目并输入需求描述。
3. 启动 `delivery_v1` 工作流。
4. 系统自动生成前后端 starter、执行基础验证并写入工作区。
5. 用户在运行页查看关键状态与验证结果。
6. 用户直接下载项目压缩包并本地运行。

### 7.3 典型设计链路

若需要强调软件工程设计能力，则可展示：

1. 用户输入需求。
2. 系统发起 `technical_design_v1`。
3. 平台输出需求规格、架构设计、前后端方案、AI 集成设计与测试计划。
4. 用户在产物中心统一查看结构化技术产物。

## 八、数据库设计

### 8.1 数据库选型说明

当前系统更适合采用“开发默认 SQLite，可选切换 MySQL”的方案：

1. `SQLite` 本地启动成本低，便于课程项目开发、联调和演示。
2. 平台核心数据具有明确结构关系，适合关系型数据库管理。
3. 若后续要扩展部署规模，可切换到 `MySQL`。
4. `Redis` 不再是本地默认依赖，仅在需要 Celery 队列化执行时启用。

### 8.2 核心实体设计

本系统核心实体包括：

1. 用户实体 `user`
2. 项目实体 `project`
3. Agent 配置实体 `agent_profile`
4. Prompt 版本实体 `prompt_template_version`
5. 流程模板实体 `workflow_template`
6. 流程步骤实体 `workflow_step`
7. 流程运行实体 `flow_run`
8. 阶段任务运行实体 `task_run`
9. 运行事件实体 `run_event`
10. 结果产物实体 `artifact`
11. 用户模型配置实体 `user_llm_config`

### 8.3 主要数据表设计

| 表名 | 关键字段 | 说明 |
|------|----------|------|
| `users` | `uid`、`email`、`display_name`、`is_admin` | 存储用户身份和权限信息 |
| `projects` | `uid`、`owner_id`、`name`、`summary`、`requirement_text`、`status` | 存储项目基础信息 |
| `agent_profiles` | `agent_code`、`display_name`、`default_model`、`enabled` | 存储角色模板元信息 |
| `prompt_template_versions` | `agent_profile_id`、`version`、`system_prompt` | 存储提示词版本 |
| `workflow_templates` | `workflow_code`、`name`、`description`、`enabled` | 存储固定工作流模板 |
| `workflow_steps` | `step_code`、`agent_code`、`depends_on`、`output_schema` | 存储工作流阶段定义 |
| `flow_runs` | `run_uid`、`project_id`、`status`、`current_stage`、`state_json` | 存储一次完整运行 |
| `task_runs` | `task_uid`、`flow_run_id`、`step_code`、`input_json`、`output_json` | 存储阶段级执行结果 |
| `run_events` | `flow_run_id`、`task_run_id`、`event_type`、`payload_json` | 存储可观测事件 |
| `artifacts` | `artifact_uid`、`artifact_type`、`title`、`content_markdown`、`content_json` | 存储用户可查看产物 |
| `user_llm_configs` | `provider_name`、`base_url`、`default_model`、`agent_runtime_overrides` | 存储账号级和角色级模型配置 |

### 8.4 表关系说明

- 一个用户可以拥有多个项目
- 一个用户最多对应一份账号级模型配置
- 一个项目可以关联多个流程运行记录
- 一个流程模板可以被多个项目重复使用
- 一次流程运行可以包含多个阶段任务运行记录和多个运行事件
- 一个项目可以生成多个产物
- 一个角色模板可以参与多个工作流阶段执行

## 九、技术栈设计

### 9.1 前端技术

- Vue 3：构建用户界面
- TypeScript：提高代码可维护性与规范性
- Vite：提升前端开发与构建效率
- Pinia：管理全局状态
- Element Plus：快速搭建管理类界面

### 9.2 后端技术

- FastAPI：构建高性能 RESTful API
- CrewAI：实现多智能体角色定义、任务组织和流程执行
- SQLAlchemy：实现 ORM 映射
- Pydantic：进行数据校验与结构化输出约束
- Thread mode：本地默认执行模式
- Celery（可选）：用于队列化异步执行
- Redis（可选）：配合 Celery 使用

### 9.3 数据与部署技术

- SQLite：本地默认持久化存储
- MySQL（可选）：正式部署或外部数据库场景下的替代方案
- Docker / Compose（可选）：统一开发与扩展部署环境

### 9.4 AI 能力接入

系统通过统一的模型调用接口接入大语言模型服务，可兼容 `OpenAI API` 格式接口，也可根据环境接入 DeepSeek、Qwen 或其他兼容服务。后端基于 CrewAI 将不同角色封装为 Agent，将结构化产出要求封装为 Task，将固定协作模板封装为 Flow，并结合项目上下文完成流程执行。

当前模型接入的特点是：

1. 用户可以配置账号默认模型。
2. 每个角色可以再单独配置自己的模型、URL 和 API Key。
3. 系统可以通过 `/models` 读取真实可用模型列表。

## 十、关键设计决策与理由

### 10.1 采用 CrewAI 作为多智能体协作运行框架

系统不自研底层多智能体编排引擎，而是采用 CrewAI 提供的 Agent、Task、Crew、Flow 能力完成流程组织。该选择能够显著降低底层实现复杂度，使团队把更多精力投入到软件工程场景设计、数据管理与结果展示上。

### 10.2 采用前后端分离 + 模块化单体后端

前后端分离便于多人协作开发，也更符合当前主流 Web 项目开发模式。后端采用模块化单体架构，能够在保证结构清晰的前提下降低部署和调试复杂度，适合课程设计周期与团队规模。

### 10.3 首版采用固定工作流模板

为控制项目复杂度，首版系统不实现完全自由的流程编排，而是采用固定工作流模板，并在其上提供角色、模型和流程启用状态的配置能力。这样既能体现多智能体协作思想，又可确保系统稳定可演示。

### 10.4 本地默认采用 `SQLite + thread`

旧方案常把 `MySQL + Redis + Celery` 作为默认前提，但这会提高课程项目启动门槛。当前系统把本地默认路径收敛为 `SQLite + thread`，可以在不启动额外中间件的情况下完成前后端联调和演示。

### 10.5 交付结果优先于内部过程

对于 `delivery_v1`，系统把“可下载、可运行”的项目结果放在第一优先级。因此平台不仅展示内部运行过程，还提供工作区物化、自动验证和项目压缩包下载能力。

### 10.6 模型配置下放给用户

相比只依赖服务端环境变量，当前系统允许用户自己配置账号默认模型和角色独立模型。这一设计更符合多智能体平台的真实使用场景，也能更直观地展示不同角色采用不同模型策略的能力。

## 十四、可行性分析

### 14.1 技术可行性

本项目使用的技术栈均为成熟技术，前后端开发门槛适中，资料丰富，适合作为课程设计项目实施。CrewAI 已提供较成熟的多智能体角色组织和流程执行能力，能够降低底层编排开发难度。项目的主要技术挑战集中在提示词设计、结构化输出约束、流程状态管理、交付验证和前端展示联动等方面，整体可控。

从项目复杂度、技术实现难度和答辩展示效果来看，该题目具有较高的课程设计适配度。通过采用成熟的 CrewAI 框架，项目团队可以减少底层重复开发，把重点放在平台设计、流程组织、交付结果与教学展示上，从而更稳妥地完成一个有特色的软件工程课程设计项目。
